1 背景

私が思うに、Carstenの目標は、Orgモードをこの素晴らしいパッケージのソースコードを読んだり改造したりする人たちだけではなく、すべてのEmacsユーザーが簡単に使えるようにすることである。そのために、品質が高くユーザ視点に立った文書を誰でも簡単に利用できるよう、Orgコミュニティによって多くの努力が続けられている。

Orgシステムのすべての機能の使用法や、ユーザー自身が機能を拡張する方法、増え続けるアドオンパッケージのいくつかを紹介しているわかりやすいガイドがOrgパッケージ自身に含まれている。このガイド(The Org Manual)はこのシステムで作業を行うことに集中している。これを補足するために、Org web pagesには多くの入門記事や、Orgを基礎としてオーガナイゼーションシステムを作るときに人々が使ったアイデアを含んだハウツー記事へのリンクが含まれている。

私はOrgを使っているが、Orgは大きなシステムである。次第に私は、Orgについての文書の一貫性、明確性、正確性を改善することは自分だけでなく他のOrgユーザーの利益になると考えるようになった。これを補助するために、そして正確性と明確性を保つために、この短いメモには私が文書を改定した際に使用したいくつかのガイドラインと規則を書いてある。他の人たちにもOrgへの貢献をする際にはこの規則を使ってほしいと願っている。

2 システム、パッケージ、モードなどの参照

もともとOrgは単一のモードであり、Orgモードが意味するものについて曖昧さはなかった。しかし、状況は変わり、CarstenはOrgを、メジャーモードといくつかのマイナーモード、それにOrgモードの基本機能の上に構築された増え続ける追加パッケージやプラグインを含んだものと今は考えている。これら全てに対する一貫した方法を見つけることは本当に難しいが、私がやろうとしていることは、次のガイドラインに従うことである(まだ完璧ではなく、スタートしたばかりである)。

  • できる限り「Org」と記述する。特に全体としてOrgに適用できるコンセプトや特徴、機能について議論する際にはそのように記述する。
  • ある特徴に固有の事柄について言及するときは、例えば、「Orgtblマイナーモード」のように、より具体的に記述する。これは、例えば、そのモードやアドオンパッケージ、プラグインをを使ってファイルを編集するときにだけ、あるコマンドが利用できるという場合である。
  • 「Org-mode」や「org-mode」よりも「Org mode」と書くようにする。これは単に「The Emacs Manual」がこの形式でモード名を記載しているという慣習を反映している(Text mode、Outline mode、Mail mode等)。
  • 同様に、Orgの機能を使って編集することが必要なファイルやバッファであることを、「Org file」または「Org buffer」と書くことで可能な限り表す。
  • OrgはEmacsコードベースの中で"org-…"を自分の名前空間を示すために使用する。これはコード中ではっきりと保たれている。

3 その他のOrg特有の慣習

より良い方法がない限り、次の慣習を適用するように努める(Carstenや他の重要なEmacsの文書に書かれている先例を参照することで正当化されると考えている。もちろん、私がそれらを作っていない場合の話である)。

  • Orgは多くのコマンドを持ち、そのうちの多くはある種のプレフィックス付き引数を取る。引数について再度言及したい場合、「プレフィクス付き引数」、「数値型プレフィクス付き引数」、または「数値型プレフィクス付き変数N」のように省略せずに書く。
  • Orgは他のEmacsパッケージと調和していたりしていなかったりする。それらのパッケージの名前を作者と保守担当者が作成した際に書くようにする。(私の考えでは)BBDBやMH-E、Rmail、VM、Gnus、CDLatexなどが該当する。
  • TODO キーワードは、Orgが定義しているか、ユーザが定義しているかにかかわらず、大文字で書く
  • Orgモードで最初から定義されている特殊な意味合いを持つタグ(例えばARCHIVE)は大文字で書く。ユーザが定義したタグ(例えば、boss、home)は小文字で書く。
  • Orgモードで最初から定義されているプロパティ(例えば、PRIORITY)は大文字で書く。ユーザ定義のプロパティ(例えば、Release)は小文字で書く
  • コンセプトインデックス中のエントリーは、他にルールがなければ、通常すべて小文字で書く。

4 org-manual.org 特有の慣習

Org gitリポジトリは、マニュアルの.org版を=doc/=ディレクトリに含んでいる。このバージョンのマニュアルに特有の指示は次のとおりである。

  • 標準のTexinfoインデックスのうち5つがOrgマニュアルで使用されている。
    #+cindex:
    コンセプトインデックス。一般的なコンセプトに使用する
    #+findex:
    関数インデックス。関数や関数風の名前に使用する
    #+kindex:
    キーストロークインデックス。キーボードコマンドに使用する
    #+pindex:
    プログラムインデックス。プログラム名に使用する
    #+vindex:
    変数インデックス。変数名に使用する
  • 1行のみの例には固定幅の領域を使用する
  • Orgの文法には、"begin_src org"ではなく、exampleブロックを使用する
  • 見出しへの文書内リンクは常に星印で開始する
  • タグ、ノードのプロパティは左右のコロンなしで表示する
  • マークアップに = … = や ~ … ~ を使うのは次の場合である。
    • ファイルや拡張子には = … = を使う。
    • Orgバッファに書かれるべきものには、= … = を使う
    • プログラミング言語内で意味を持つトークンには ~ … ~ を使う

5 その他

  • コンセプトとキーに対して標準的なTexinfoインデックスのうち2つのみが使われる。これには次のような意味合いを含んでいる。
    • コマンドは名前ではなくキーを用いて説明することが望ましい。
    • Texinfoコマンドのうち、@varや@defoptionは使用しない。これらが示す事柄については、ユーザーがカスタマイズグループを閲覧することが望ましいからである。もし、例えば、変数名について述べる必要があるならば、「変数@code{org-startup-folded}」のように記述する。
    • コンセプトインデックス中のエントリは通常、特に他のルールが指示されていない場合、全て小文字である。
  • Org文書は米語で書かれている。自分は米語が少し苦手であるが、米語で生活している。
  • Orgはいくつかの複合語を使っている。それらを分離するか、ハイフンを入れるか、それとも複合語のままで行くかどうか悩む代わりに、大抵の場合原文のままにし、スペルチェッカをこの選ばれた標準に対応させ、それ以上気にしないことにしている。
  • 私は定期的にスペルチェッカをかける。Aspellはうまく動き、Texinfoコマンドを無視するフィルタ機能を持っている(しかし、困ったことにispell.elではうまく行かないため、コマンドラインから実行している)。私はOrg専用のAspell設定ファイル(米語辞書の使用と複合語に対するルール等)と自分が使用したより詳細である程度任意性を持ったルールを、関連語と代替語のファイルとともに保存している。

  • Orgを使うための壁は低い。必要なのは次のことだけと思われる。

    • Text mode、またはその派生モードを使用できる。
    • より生活を組織的にしたいという動機を持っている。

    それゆえに、関連する文書を書いてみれば、様々な人々に読んでもらえる。
